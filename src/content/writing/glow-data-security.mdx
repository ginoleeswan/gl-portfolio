## The data is the sensitive part
Glow stores who you care about, what you wrote to them, and when. If one user could ever read
another's records, a nice relationship app turns into a privacy problem quickly. So the question
I cared about wasn't how to fetch a user's data — it was where the line sits that stops one user
from reading another's. I didn't want that line to be my React code.

## Two rules
Isolation lives in the database. Every user-owned table (people, events, glows) uses row-level
security tied to `auth.uid()`. Postgres won't return another user's rows regardless of what the
client asks for, so if I write a bad query in the front end, RLS still holds. Putting isolation
in application code means every new feature is one missing `where user_id = …` away from a leak,
and I'd rather not rely on remembering it each time.

Secrets stay on the server. The OpenAI calls and the Resend emails run in Supabase Edge
Functions. The client calls my function and the function holds the key, so nothing secret ends up
in a bundle someone can read in dev-tools. A key in a client-exposed env var is already a
published key.

## The bug that made the point
RLS protects the database, not a cache in the browser, and that's where it bit me. After signing
out and into a different account, the TanStack Query cache still held the previous session's
data, so the new user briefly saw records that weren't theirs. The database was fine; the stale
copy was entirely client-side.

I fixed it by scoping the cache to the session: I listen for Supabase auth-state changes and clear
the query cache when the session changes, so signing out doesn't leave a warm cache for whoever
logs in next.

The takeaway was that one safe layer isn't enough. The database did its job and I still had a
leak, because the cache had its own lifetime. Anything that holds user data needs its own answer
for what happens at logout.

## Where it ended up
RLS in Postgres decides who can read what. Edge Functions hold the secrets. The client cache is
treated as throwaway and tied to the session. If the app gets something wrong, your data still
shouldn't land on someone else's screen.
